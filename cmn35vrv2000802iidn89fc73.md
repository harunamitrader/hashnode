## まず前提

- `Git` は、変更履歴を管理する仕組み
- `GitHub` は、その履歴を共有しやすくするサービス

つまり、関連用語の多くは「変更を安全に進めるための言葉」です。

## リポジトリ

リポジトリは、**ファイルと変更履歴をまとめて置いておく場所**です。

難しく聞こえますが、「このプロジェクト一式の保管庫」くらいで大丈夫です。

たとえば、

- Web サイトのファイル一式
- 記事の原稿一式
- アプリのソース一式

のようなまとまりを入れておく箱だと思えば分かりやすいです。

## コミット

コミットは、**変更の一区切りを記録すること**です。

「ここまで直したので、いったん保存しておく」というチェックポイントに近いです。

ただの上書き保存との違いは、あとで「この時点で何を変えたか」を見返しやすいことです。

## ブランチ

ブランチは、**本流とは別に作る作業用の分かれ道**です。

いきなり本番の内容を触るのではなく、別ラインで試して、よければ後で取り込むために使います。

感覚としては、

- 本番原稿を直接直すのではなく
- コピーを作ってそこで修正し
- 問題なければ本番へ戻す

に近いです。

## マージ

マージは、**別の場所で進めた変更を本流へ合流させること**です。

ブランチで作業した内容を、「これでOK」と判断して本体へ戻す場面で出てきます。

## クローン

クローンは、**リポジトリを自分の手元にコピーしてくること**です。

GitHub 上にあるものを、自分の PC でも触れるように持ってくるイメージです。

## Fork

Fork は、**他人のリポジトリを自分のアカウント側へコピーすること**です。

クローンが「手元の PC に持ってくる」なら、Fork は「GitHub 上の自分の場所に分けてもらう」イメージです。OSS を触るときによく出てきます。

## local / remote

この 2 つはセットで出てきます。

- `local` = 自分の PC 上の作業場所
- `remote` = GitHub など、共有先の場所

つまり、ローカルで作業して、必要になったらリモートへ送る、という流れです。

## プッシュ

プッシュは、**手元で行った変更を GitHub 側へ送ること**です。

ローカルで直しただけでは、他の人には見えません。GitHub に反映して初めて共有しやすくなります。

## プル

プルは、**GitHub 側にある最新の変更を自分の手元へ取り込むこと**です。

チーム作業では、自分だけ古い状態のまま進めないために大事です。

## プルリクエスト

プルリクエストは、**この変更を取り込んでください、とお願いする仕組み**です。

略して `PR`、日本語だと `プルリク` と呼ばれることが多いです。

これはかなり重要で、

- どこを変えたのか見せる
- 他の人に確認してもらう
- 問題なければ取り込む

というレビューの流れを作ります。

## main

`main` は、**そのリポジトリの本流として使われることが多いブランチ名**です。

昔は `master` がよく使われていましたが、今は `main` が一般的です。ブランチの説明とセットで、「最終的にここへ戻す先」と理解すると分かりやすいです。

## コンフリクト

コンフリクトは、**変更どうしがぶつかること**です。

たとえば同じ場所を別の人がそれぞれ直していたとき、「どちらを採用するのか決めないといけない」状態になります。

怖く見えますが、要するに「修正内容が競合したので整理が必要です」という意味です。

## gitignore

`gitignore` は、**Git で管理しないファイルを決めるための設定**です。

たとえば、一時ファイルや個人のローカル設定、外に出したくないファイルなどを追跡対象から外すときに使います。

## README

`README` は、**そのリポジトリの説明書**です。

「これは何のプロジェクトか」「どう使うのか」「最初に何を読めばいいのか」が書かれていることが多く、GitHub で最初に目に入る重要なファイルです。

## Issue

`Issue` は、**課題、相談、修正依頼を書く場所**です。

「ここを直したい」「この不具合がある」「この機能を追加したい」といった話を、チームや OSS の参加者どうしで整理するために使います。

## 用語を全部覚えなくてもいい

非エンジニアが最初に押さえるなら、このくらいで十分です。

- リポジトリ = プロジェクト一式の箱
- コミット = 変更のチェックポイント
- ブランチ = 別作業ライン
- マージ = 合流
- プルリク = 変更確認のお願い
- local / remote = 手元と共有先
- main = 本流
- gitignore = 管理しないファイルの指定
- README = 説明書
- Issue = 課題や相談を書く場所

この 5 つがざっくり分かると、GitHub 上の会話がかなり読みやすくなります。

## まとめ

Git や GitHub の関連用語は、最初は外国語みたいに見えます。でも実際は、「変更を安全に進めるための役割名」が多いです。

全部を正確に暗記しなくても、まずは

- どこに置くのか
- どこで直すのか
- どう共有するのか
- どう本流に戻すのか

が分かれば十分です。

それだけでも、開発チームとの会話はかなり追いやすくなります。
